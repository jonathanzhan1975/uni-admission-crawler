# 方案 A POC — 需求规格说明书（SRS）

> 文档目的：作为 POC 阶段的**唯一事实源**，进入开发前需用户签字确认。开发完成后逐条对照验收。
>
> 范围：复旦大学 + 上海交通大学 + 清华大学 招生相关消息每日聚合推送。

---

## 1. 业务背景与目标

### 1.1 问题陈述
用户需要每日跟踪国内 985 高校的招生动态（含本科招生、研究生招生、新增专业、招生政策等），目前依赖人工浏览各高校官网/招办，耗时且易遗漏。

### 1.2 POC 目标
在 3 所代表性大学上，验证"自动抓取 → 智能过滤 → 每日推送"全链路在工程、成本、内容质量三个维度均可行，作为后续扩展到 39 所 985 的技术决策依据。

### 1.3 非目标（明确不做）
- **不做** 历史回溯（只抓今天起的新消息）
- **不做** Web UI / 用户管理 / 多租户
- **不做** 微信公众号抓取
- **不做** 高校间对比、排名、统计分析
- **不做** 推送内容的双向交互（评论/转发等）

## 2. 用户画像与典型场景

| 用户角色 | 描述 | 关心的事 |
|---|---|---|
| 高三学生家长 | 关注本科招生政策、新专业 | 招生简章、专业新增、强基/综评 |
| 考研学生 | 关注研招办公告 | 报名、复试、调剂信息 |

POC 阶段服务于**单一用户**，无需考虑权限/订阅偏好。

### 3.2 数据源 SLA 假设
- 每个数据源每日新增 ≤ 50 条（超出告警，提示路由选错）
- 列表页加载时间 ≤ 10 秒
- 容许任意单一数据源在 24h 内不可用，不影响其他源推送

### 3.3 数据源实现注意事项

- 清华本科招办 `join-tsinghua.edu.cn` 使用静态 HTML 列表，当前有效栏目为 `/zygg.htm`（重要公告）；`/zsxx.htm` 为 404，`/info/1033/` 为系统提示页，不应作为列表入口。
- 上交本科招办 `admissions.sjtu.edu.cn` 为 Nuxt 站点。首屏 HTML 已包含 `window.__NUXT__` SSR 注水数据，并在页面中直接渲染"重要资讯"列表（栏目 ID `3810134`）；Nuxt chunk 实测暴露同源 JSON API `POST /rnapi/newsList`，当前 S-07 直接调用该 API，避免引入 Playwright 影响 NFR-1 性能预算。当前仅抓"重要资讯"栏目，"招生政策""招生计划""报考指南"等其他栏目可在方案 B 前补全。

## 4. 功能需求（FR）

### FR-1 抓取
- **FR-1.1** 系统应每日**北京时间 09:00** 自动执行一次完整抓取
- **FR-1.2** 每个数据源最多抓取列表页**前 30 条**
- **FR-1.3** 抓取失败应自动重试 3 次，间隔指数退避（10s, 30s, 90s）。**注：推送层重试策略不同，见 FR-6.7**
- **FR-1.4** 任一数据源连续 3 次失败后跳过本次，不阻塞其他源
- **FR-1.5** 原始 HTML/RSS 内容应落盘归档至 `raw/<YYYY-MM-DD>/<source_id>_<sha>.html`，保留 30 天

### FR-2 标准化
- **FR-2.1** 所有抓取结果应转换为统一 Item 结构：
  ```
  {
    item_id: str          # = sha256(canonical_url)，主键
    university: str       # 中文全称
    source_id: str        # S-01 ~ S-07
    source_name: str      # "本科招办" / "研招办" / "教务处" / "主新闻"
    title: str            # 已去除空白字符
    url: str              # canonical 后的绝对 URL
    pub_date: str         # ISO8601，无时区视为 UTC+8
    summary: str | null   # 列表页摘要，可空
    fetched_at: str       # ISO8601 UTC
  }
  ```
- **FR-2.2** 缺失 `title` 或 `url` 的 Item 应被丢弃并记录警告
- **FR-2.3** `pub_date` 缺失时使用 `fetched_at`，并标记 `date_inferred: true`

### FR-3 分类（仅对 needs_classification=true 的源）
- **FR-3.1 关键词预筛**：标题 OR 摘要命中以下任一关键词通过：
  ```
  招生 | 录取 | 报考 | 本科专业 | 新增专业 | 招生简章 | 推免 |
  强基 | 综评 | 选拔 | 转专业 | 高考 | 复试 | 调剂 | 报名
  ```
- **FR-3.2** 预筛通过的 Item 送 LLM 终判，输出 `is_admission: bool` + `reason: str`
- **FR-3.3** 仅 `is_admission=true` 的 Item 进入推送
- **FR-3.4** 预筛**未通过**的 Item 直接丢弃（不消耗 LLM 配额）
- **FR-3.5** LLM API 不可用时降级为"仅关键词"模式，并在推送底部标注降级状态

### FR-4 去重
- **FR-4.1** 使用 SQLite 持久化已推送 Item，主键 = `item_id`
- **FR-4.2** 已存在于 SQLite 中的 Item 不再推送
- **FR-4.3** 推送成功后再写入 SQLite（**原子性**：推送失败则不入库，下次重试）
- **FR-4.4** SQLite 文件应随 Git 提交，保证 GitHub Actions 跨次运行有状态

### FR-5 渲染
- **FR-5.1** 输出 Markdown 格式日报，结构：
  ```
  # 📅 高校招生动态 · YYYY-MM-DD
  
  > 共 N 条新消息（来自 X 所高校）
  
  ## 🟦 复旦大学
  ### 本科招办
  - [<title>](<url>) · YYYY-MM-DD
  ### 研招办
  - [<title>](<url>) · YYYY-MM-DD
  
  ## 🟩 上海交通大学
  ...
  
  ## 🟨 清华大学（经 AI 分类）
  ...
  
  ---
  *分类降级状态：正常 | LLM 不可用*
  *本次运行 ID：<github_run_id>*
  ```
- **FR-5.2** 当某高校今日 0 条更新时，仍显示该高校标题，附"今日无更新"
- **FR-5.3** 总条数为 0 时仍发送推送（避免用户怀疑系统挂了）

### FR-6 推送（多渠道）

- **FR-6.1** 系统应同时推送到以下渠道，**任一渠道独立失败不影响其他渠道**：
  | 渠道 ID | 渠道 | 用途 | POC 当前状态 |
  |---|---|---|---|
  | CH-1 | 飞书群机器人 Webhook | 桌面/团队协作场景 | **暂关闭**（待用户配置 webhook 后开启）|
  | CH-2 | Server 酱 (sct.ftqq.com) | 手机微信即时查看 | **启用** |

> 当前 POC 仅 CH-2 启用。CH-1 实现完整保留并通过 `config.yaml` 中 `enabled: false` 关闭，后续配齐 webhook 后切换 `enabled: true` 即可启用，无需改代码。

- **FR-6.2 飞书推送（CH-1）**
  - 使用 webhook 推送，消息类型 `interactive` 卡片或 `text`
  - 单条 > 30KB 时按高校切分多条
  - 重试策略遵循 FR-6.7

- **FR-6.3 Server 酱推送（CH-2）**
  - 使用 `https://sctapi.ftqq.com/<SENDKEY>.send` 接口（JSON 提交，`Content-Type: application/json; charset=utf-8`）
  - 请求字段：`title`（≤ 32 字）、`desp`（Markdown 正文，≤ 32KB）
  - title 格式：`📅 高校招生动态 · MM-DD（共 N 条）`
  - `desp` 复用同一份 Markdown 渲染产物
  - **Server 酱免费版限频 5 条/天**（不是 5 条/分钟）；月 500 条上限
  - POC 每日 1 次定时推送，给手动验收/失败重跑预留 ~4 条/天 buffer
  - 返回 JSON 中 `code != 0` 视为失败，按 FR-6.7 决定是否重试

- **FR-6.4 渲染共享**：两渠道共用同一份 Markdown，**避免双份维护**。差异仅在元数据（标题截断、卡片包装）

- **FR-6.5 失败处理**：
  - 单渠道失败：本渠道按 FR-6.7 重试用尽后写入 `failed/<YYYY-MM-DD>_<channel>.md`，**不**影响其他渠道
  - 全部启用渠道失败：`exit 1`，让 GitHub Actions 标红
  - 任一渠道成功：dedup SQLite 入库（避免次日重复推送已成功送达的内容）

- **FR-6.6 渠道开关**：通过 `config.yaml` 中 `channels.<id>.enabled = true|false` 单独开关，便于演练（重复 §4 章节中已有约定，此处保留对照）

- **FR-6.7 推送层重试分流**（**与 FR-1.3 抓取层不同**）：

  | 错误类型 | 重试 | 说明 |
  |---|---|---|
  | HTTP 5xx | **3 次**，10/30/90s 退避 | 服务端临时故障，重试有意义 |
  | 网络错误 / 超时 | **3 次**，10/30/90s 退避 | 同上 |
  | HTTP 4xx（400/401/403/404）| **0 次**，立即失败 | 重试无法把 4xx 变 2xx，且烧外部配额 |
  | HTTP 200 但业务 `code != 0` | **0 次**，立即失败 | 业务错误（鉴权/参数/限额），重试无意义 |
  | HTTP 429 或限额业务码 | **0 次**，立即失败 | 错误日志须含 `error_kind=quota_exhausted` 字段，便于告警识别 |

  设计依据：Server 酱免费版限额仅 5 条/天，盲目 3 次重试 4xx 会无效消耗 4 条额度且不会成功，必须区分对待。

## 5. 非功能需求（NFR）

| ID | 类别 | 要求 |
|---|---|---|
| NFR-1 | 性能 | 单次完整运行 ≤ 90 秒（不含 LLM 调用），含 LLM ≤ 180 秒 |
| NFR-2 | 成本 | 月运营成本 ≤ ¥10（仅 LLM API） |
| NFR-3 | 可观测 | 每次运行输出结构化日志（JSON），关键字段：source_id, fetched_count, deduped_count, classified_count, pushed_count, errors |
| NFR-4 | 可重现 | 所有依赖 lock（requirements.txt 或 poetry.lock） |
| NFR-5 | 安全 | 飞书 Webhook URL、Server 酱 SENDKEY、Google API Key 仅通过 GitHub Secrets 注入，不入库 |
| NFR-6 | 鲁棒 | 任一数据源解析变更（如 HTML 改版），系统应跳过该源继续运行，并在日志告警 |
| NFR-7 | 合规 | 抓取频率 ≤ 1 次/天/源，遵守 robots.txt，User-Agent 标识为善意爬虫 |

- [ ] **A6** 源失败演练（手动 ban 复旦域名）后，上交+清华仍正常推送（CH-1 暂关闭，源隔离演练在单渠道下验证）

## 8. 风险登记册

| 风险 | 概率 | 影响 | 应对 |
|---|---|---|---|
| 复旦招办 HTML 改版 | 中 | 单源失效 | NFR-6 跳过 + 告警，2 日内修复 |
| RSSHub 公共实例被 ban | 中 | 上交+清华失效 | 切换到 Vercel 自部署（已预备方案） |
| 清华分类准确率不达标 | 中 | 误推/漏推 | 调整关键词 + prompt，增大金标集 |
| GitHub Actions 配额耗尽 | 低 | 推送中断 | 公开仓库无配额限制 |
| 飞书机器人被踢 | 低 | 飞书渠道失败 | Server 酱兜底 + FR-6.5 落盘 |
| Server 酱免费版被限频 | 低 | 微信渠道失效 | POC 每日 1 次远低于上限；监控 `code` 字段，限频时延后重试 |
| 微信屏蔽方糖服务号 | 低 | 微信渠道失效 | 飞书兜底；用户可手动重新关注 |
| SENDKEY 泄露 | 中 | 他人可发垃圾消息到你微信 | GitHub Secret + .gitignore + 定期轮换 |
