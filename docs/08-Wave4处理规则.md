# Wave 4 SPA/API 处理规则

为保证项目 ROI 并避免无意义的沉没成本，Wave 4 实施遵循以下硬性规则：

## 1. 单所探测时间盒 (Timebox)
- **上限：60 分钟**。
- 如果在 60 分钟内无法找到稳定、免认证、返回 JSON 的 API，必须立即停止。

## 2. 尝试上限 (Max Attempts)
严禁进行发散式探测，仅限以下三种路径：
1. **SSR 注水提取**：尝试一次 `window.__NUXT__` 或 `__NEXT_DATA__` 解析。
2. **API 路径探测**：尝试 2-3 个基于现有经验（如 `/rnapi/newsList`）或开发者网络观察到的明显 API 路径。
3. **失败即停**：一旦以上路径不通，立即标记为 `known_degraded`，不启动 v4/v5/v6 方案。

## 3. `known_degraded` 处理规范
对于探测失败的学校：
1. **文档更新**：在 `docs/sources/<ns>.md` 中增加 `## 实施警告` 章节，写明放弃原因（如：Nuxt SPA 且无公开 API）。
2. **配置隔离**：在 `config.yaml` 中注释掉该 Source 项，保留 ID 占位但停止抓取。
3. **未来路线**：在文档中增加 `## 未来工作` 章节，记录待尝试方向（如 Playwright、SSR 注水等），留待后续专项攻克。

## 4. 严禁跨 Session 持续尝试
严禁在同一所学校上跨多次 Session 持续投入时间。探测结果一旦记录，除非有重大技术突破或官方接口变更，否则不再回溯。
