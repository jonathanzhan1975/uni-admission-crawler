## 0. 安全与合规声明（最高优先级）

本方案及 Phase 1 审计脚本属于**公开网页可用性 / 内容覆盖审计**，目的是判断各高校官方招生信息源是否能被正常订阅与解析。

**明确属于本范围（允许）**：
- 对**已公开**的高校官网首页 / 招办公告列表页发起 GET
- 对**已公开**的 RSSHub 路由发起 GET
- 对**已文档化**的招办 SPA 后端 JSON API 发起 POST（如 SJTU `/rnapi/newsList`，仅传公开列表参数）
- 单 IP 低频访问（间隔 ≥ 5-10 秒），与普通浏览者行为一致
- 标识真实 User-Agent：`uni-admission-crawler/0.1 (audit; +https://github.com/jonathanzhan1975/uni-admission-crawler)`

**明确不属于本范围（禁止，任何工具/脚本/手工均不可）**：
- ❌ 漏洞扫描 / 渗透测试 / 安全测试
- ❌ 目录爆破 / 路径枚举 / 敏感文件探测
- ❌ 登录尝试 / 凭证爆破 / 绕过认证 / 提权
- ❌ SQL 注入 / XSS / SSRF / 任何 OWASP 类测试
- ❌ 对任何端点的高频压力 / 拒绝服务行为
- ❌ 收集 / 留存任何个人信息 / 学生 ID / 联系方式
- ❌ 抓取需要付费 / 注册 / 鉴权才能访问的内容

**遇到反爬 / WAF（如 412 / 403 / 验证码）时的标准动作**：
立即标记该源为 `waf_blocked`，**不尝试任何绕过手段**，转入「需要管理员授权 / 联系官方开放数据接口」工作流。

> 本声明对方案文档中所有阶段（Phase 1-4、Wave 1-4）和所有自动化脚本均生效。如审计脚本被任何安全/合规系统标记，立即停止运行，按本节边界自查后再恢复。

## 3. Phase 1 — 自动化源审计

### 3.1 输出物

`docs/06-源审计报告.md`：39 行 × 10 列表格，作为 Phase 3 实施的唯一输入。

### 3.2 审计脚本规格（给 codex 实现）

新建一次性脚本 `scripts/audit_sources.py`：

**输入**：
- 39 所 985 名单（硬编码或读 `docs/00-方案总览.md` §3.1/3.2 表格）
- RSSHub 自部署实例 URL（`https://rsshub-uni-admission.vercel.app`）
- 招生关键词列表（复用 `classifier.KEYWORDS`，但不含 EXCLUDED_KEYWORDS）

**对每所大学执行**：

```
Step 1: 查 RSSHub namespace 是否存在
  - 已知 28 所有 namespace（见 docs/00-方案总览.md §3.1）
  - 未知 11 所跳过 RSSHub 部分，直接进 Step 4

Step 2: 列出 namespace 下所有路由
  - GET https://api.github.com/repos/DIYgod/RSSHub/contents/lib/routes/<ns>
  - 解析 .ts 文件名得到路由列表
  - 优先级选路：含 zsb/admission/yzb/news 关键词的优先

Step 3: 自动拉取 + 判定覆盖度
  - 对每所最多试 3 条路由（按 Step 2 优先级）
  - GET https://rsshub-uni-admission.vercel.app/<ns>/<route>
  - 解析 RSS 取前 10 条 item title + summary
  - 判定算法：
    - 拒否条件先行：含 EXCLUDED_KEYWORDS（研究生口径）→ 该 item 不计入分子
    - 命中 KEYWORDS（招生口径）→ +1 分子
    - 分母 = 总 item 数
    - 比率 > 50%   → 评级 A（纯招办）
    - 比率 10-50%  → 评级 B（含招生需 LLM）
    - 比率 < 10%   → 评级 C（几乎不含招生）
  - 取最高评级的路由作为代表

Step 4: 找招生官网 URL（仅 P0/P1 需要）
  - 用 DuckDuckGo 或 Bing 搜索 API（如可用）
  - query: "<校名> 本科招生网"
  - 取第一条 .edu.cn 域名结果
  - 不可用则留空，标记为 "manual_needed"

Step 5: 探测招办 CMS 类型（仅 P0/P1 需要）
  - GET 招办首页 HTML（如 Step 4 拿到）
  - 嗅探指纹：
    - 含 "wp_article_list" → webplus
    - 含 "data-n-head-ssr" 或 "_nuxt/" → nuxt-spa
    - 含 "react" + "__NEXT_DATA__" → next-spa
    - 含 <table> 或简单 <ul><li> → static
  - 不明确 → "unknown"

Step 6: 计算优先级
  - rsshub 评级 C → P0（必自写）
  - rsshub 评级 B → P1（自写优先，RSSHub 凑合也行）
  - rsshub 评级 A → P2（暂缓自写）
  - 无 RSSHub namespace + 无 bkzs URL → 标 BLOCKED，人工介入
```

**输出**：
- `docs/06-源审计报告.md`：表格形式
- `docs/06-源审计原始数据.json`：所有抓取的 raw item titles，便于人工抽检

### 3.3 审计脚本质量要求

- 全程**只读**：不写代码、不写 config、不触发推送
- 安全边界：这是**公开网页可用性审计**，仅访问高校和 RSSHub 已公开的 GET/POST 页面/API；不做漏洞探测、目录爆破、登录尝试、绕过认证、压力测试或任何安全测试行为
- 失败容错：单所失败不影响其他所（参考 fetcher I-1 约定）
- 限速：访问 Vercel RSSHub / 招办页面间隔默认 8s，最低建议 5-10s，避免触发安全系统误报或对外站造成压力
- User-Agent 同 `crawler/utils/http.py`：`uni-admission-crawler/0.1 (audit)`
- 耗时预算：39 所 × ~15s/所 ≈ 10 分钟，远低于 NFR-1 180s
- 脚本必须支持：
  - `--dry-run`：只打印计划访问目标，不发起网络请求
  - `--no-search`：禁用搜索引擎查询，只使用维护清单中的 URL
  - `--max-schools N`：限制本次审计学校数量，便于分批跑

## 5. Phase 3 — 实施 4 波次

### Wave 1（5/15-5/17）：RSSHub 直接复用（28 所）

- 输入：Phase 1 中评级 A/B 的源
- 工作量：每所只改 `config/config.yaml` 加 1 个 source 块
- 测试：每所加 1 个 `@pytest.mark.live` 测试
- 阻塞性：低（无新代码）

### Wave 2（5/19-5/21）：Webplus CMS 自写（~15 所）

- 抽象 `WebplusFetcher` 基类（重构 FudanAoFetcher）
- 对 Phase 1 探测为 webplus 的每所写子类
- 复用现有 LIST_PATHS / selector 多套兼容机制

### Wave 3（5/22-5/24）：SPA / API 自写（~5-8 所）

- 每所单独调研 XHR API（参考 SJTU 套路）
- 找不到 API 的，2 选 1：
  - 上 Playwright（+200MB 镜像 + 30s 单次运行，仍 < NFR-1 180s）
  - 退回 RSSHub 凑合 + 标记 known_degraded

### Wave 4（5/25）：容量复测 + 39 所验收启动

- Server 酱 desp 长度复测：39 所合并预期日推送 50-150 条 → Markdown 可能超 32KB
  - 应对：按地区分组（在京/在沪/华东/华中/华南/华东北/西部）渲染多个二级标题；如仍超 32KB 触发 split_by_size 自动切分
- Vercel RSSHub 容量复测：28 路由 × 1 次/天 = 28 请求/天，免费层千万级请求绰绰有余
- Gemini Flash 复测：预计 200-300 LLM 调用/天，远低于 1500 RPD
- GitHub Actions 时间复测：fetcher 串行 39 所 × ~5s = ~3 分钟，需改为**并发抓取**（ThreadPoolExecutor）压到 < NFR-1 180s

> ### v2 修复点（基于第一版抽检发现）
>
> #### 修复 1：区分「服务不可用」与「无招生内容」
> - 现状：HTTP 503 → 0/0 命中 → 评级 C → P0
> - 修复：HTTP ≥ 500 时该路由标 `unavailable`，**不计入**评级分母；如该校所有路由都 unavailable，整校 `rsshub_status=unavailable`，评级写 `?` 而非 C
>
> #### 修复 2：代表路由选择优先级
> - 现状：字母序前 N 条
> - 修复：按招生相关度排优先级：含 `zsb / admission / zs / news` 关键词 > 含 `jwc / yzb / gs` > 其他。每所先试 5 条，跳过 503 取下一条
>
> #### 修复 3：bkzs URL 主动校验
> - 现状：找到一个 URL 就写报告（结果北大 `gotopku.cn` 完全不存在 / DNS 失败）
> - 修复：每个找到的 URL 必须 `curl -A "Mozilla/5.0" --max-time 8` 实测：
>   - 200/301/302 → 接受
>   - 412/403 → 标 `waf_blocked`，URL 仍写但备注需手工处理（**不尝试任何绕过**）
>   - 4xx 其他 → 弃用，尝试备选 URL
>   - 000 / DNS 失败 → 弃用，尝试备选
> - 备选 URL 列表（按命中率排序）：`admission.{ns}.edu.cn` → `bkzs.{ns}.edu.cn` → `zsb.{ns}.edu.cn` → `zs.{ns}.edu.cn` → `zhaosheng.{ns}.edu.cn`
